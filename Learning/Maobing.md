# 一、计算机视觉四大任务（视觉的四大金刚）

工业视觉项目中，绝大部分需求可以归纳为四大类：**检测、定位、分类、测量**。理解它们的定义、适用场景和常用算法，是做方案选型的基础。

## 1.1 任务定义与对比

| 任务 | 英文 | 核心目标 | 输出内容 | 典型场景 | 评价指标 |
|------|------|---------|---------|---------|---------|
| **分类** | Classification | 判断图像/区域属于哪一类别 | 类别标签 + 置信度 | 缺陷分类（好品/坏品）、物体识别 | Accuracy, Precision, Recall, F1 |
| **检测** | Detection | 找出图像中所有目标实例的位置和类别 | 边界框 + 类别标签 + 置信度 | 瑕疵检出、多目标定位、行人检测 | mAP, IoU, Recall, Precision |
| **定位** | Localization | 精确确定目标在图像中的位置和姿态 | 位置坐标（点/框/轮廓）+ 角度 | 对位引导、抓取定位、PCB 对位 | 偏差距离（pixel/mm）、角度误差 |
| **测量** | Measurement | 量化目标的几何尺寸或物理属性 | 长度、角度、面积、直径等数值 | 尺寸检测、间隙测量、孔径测量 | 测量精度（μm）、重复性（GR&R） |

## 1.2 任务之间的关联

```
分类（是什么）
   │
   ▼
检测（在哪里 + 是什么）
   │
   ├──→ 定位（精确位置与姿态）
   │
   └──→ 测量（几何尺寸量化）
```

- **分类**是最基础的任务，通常作为其他任务的前置或辅助模块。
- **检测** = 定位 + 分类，同时解决"在哪里"和"是什么"。
- **定位**强调**空间精度**，输出坐标系下的精确位置，常用于机器人引导。
- **测量**强调**尺寸精度**，输出物理单位（mm/μm）的度量值，需要标定。

## 1.3 各任务的常用算法

| 任务 | 传统算法（HALCON） | 深度学习算法 |
|------|------------------|-------------|
| **分类** | 灰度直方图 + 贝叶斯分类、SVM | ResNet、EfficientNet、Vision Transformer |
| **检测** | 差分法、Blob 分析、模板匹配 | YOLO 系列、Faster R-CNN、DETR |
| **定位** | NCC 模板匹配、形状匹配、几何拟合 | 关键点检测（HRNet）、分割+求中心 |
| **测量** | 亚像素边缘提取 + 卡尺工具、轮廓拟合 | 分割 → 边缘提取（作为深度学习的下游） |

## 1.4 工业场景中的常见组合

实际项目中，四大金刚往往是组合使用的：

- **检测 + 测量**：先检测出缺陷区域，再测量缺陷的尺寸是否超标
- **定位 + 检测**：先定位产品位置（粗定位），再在局部区域检测细微缺陷
- **分类 + 定位**：先判断产品型号（分类），再根据型号切换不同的定位模板

# 二、ONNX Runtime 转 TensorRT 的 Top-K 问题

## 2.1 背景：为什么需要 ORT → TRT 转换

| 概念 | 全称 | 定位 | 特点 |
|------|------|------|------|
| **ONNX Runtime (ORT)** | Open Neural Network Exchange Runtime | 跨平台推理引擎 | 支持多种硬件后端，兼容性好，通用部署 |
| **TensorRT (TRT)** | NVIDIA TensorRT | GPU 专用推理优化引擎 | 针对 NVIDIA GPU 深度优化，推理速度更快 |

工程流程通常是：训练（PyTorch/TF）→ 导出 ONNX → ORT 验证 → **转换 TRT** → 部署到产线 GPU 工控机。TRT 可以显著提升推理速度（通常 1.5~3x），但转换过程中常遇到算子兼容性问题，**Top-K** 就是其中最常见的一个。

## 2.2 什么是 Top-K 操作

Top-K 从模型的输出向量（如 1000 个类别的概率分数）中，找出概率最高的 K 个结果，返回两样东西：

| 返回值 | 含义 | 类型 |
|--------|------|------|
| **Values（值）** | 这 K 个最高的概率分数 | float |
| **Indices（索引）** | 这 K 个分数对应的类别编号 | int（int64 / int32） |

> 举例：分类模型输出 1000 个分数，取 K=5，得到「概率最高的前 5 个类别分别是什么，置信度是多少」。

## 2.3 ORT 与 TRT 的核心冲突

ORT 和 TRT 对 Top-K 的实现哲学不同，导致转换时出现不兼容：

| 方面 | ONNX Runtime (ORT) | TensorRT (TRT) | 冲突点 |
|------|-------------------|----------------|--------|
| **K 值的动态性** | 支持 K 值可变（可作为动态输入） | **要求 K 值固定**（作为编译期常量属性） | ❗ 最常见的问题。ONNX 中 TopK 的 K 如果是动态的，TRT 直接报错 |
| **输出排序** | 默认不保证排序 | 默认按分数从高到低排序 | 后续节点依赖特定顺序时，结果可能不正确 |
| **索引数据类型** | int64 或 int32 | 偏好 int32 | int64 → int32 隐式转换可能截断或报错 |
| **算子版本** | 版本众多（1, 9, 11, 12...） | 对特定版本支持度有限 | 老版本 ONNX TopK 的行为与 TRT 预期不符 |

### 典型报错信息

```
[TRT] ModelImporter.cpp: error: TopK with dynamic K value is not supported
```

这句话的含义是：TRT 在编译时需要确定 K 的具体数值，但你的 ONNX 模型中 TopK 节点的 K 是某个变量计算得来的（动态值），TRT 无法处理。

## 2.4 其他常见表现

| 现象 | 原因 |
|------|------|
| 转换直接失败 | K 值为动态，TRT 无法编译 |
| Top-1 / Top-5 精度有细微差异 | 输出排序顺序变化，或数据类型截断 |
| 推理速度不达预期 | TRT 为兼容动态 Top-K 未选用最高效实现 |
| 索引越界 / 内存错误 | int64 vs int32 不匹配，后续 Gather 节点访问错误 |
| 结果完全错误 | 排序属性（sorted=0/1）不一致 |

## 2.5 解决方案（按推荐优先级排序）

### 方案一：后置处理（推荐）

```
┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│   TRT 引擎   │ →   │ 概率向量输出 │ →   │ CPU 后处理   │
│  (不含Top-K) │     │ (完整向量)   │     │ 做 Top-K     │
└──────────────┘     └──────────────┘     └──────────────┘
```

**做法**：让 TRT 只负责计算到最后一个完整的概率向量，**把 Top-K 操作移出 TRT 引擎**，在推理引擎外部用 CPU 代码（如 `argpartition`）做 Top-K。

**优点**：
- 彻底避免转换问题
- K 值完全灵活，可动态调整
- 不损失精度
- 最简单、最可靠

### 方案二：固定 K 值

- 用 `onnx-simplifier` 等工具化简模型，尝试将动态 K 折叠为常量
- 修改 ONNX 导出代码，将 TopK 的 K 设为 `np.int64(K)` 常量节点

### 方案三：显式设置排序属性

- 导出 ONNX 时设置 `sorted=1` 或 `sorted=0`
- 使用 `trtexec` 时指定排序策略，确保 ORT 和 TRT 行为一致

### 方案四：模拟替代（K=1 时）

- 用 `ArgMax` + `Gather` 组合模拟固定 K=1 的 Top-1
- 避免了 TopK 算子本身

### 方案五：更换模型导出方式

- 有些框架（如 `torch-tensorrt`）可以直接从 PyTorch 导出 TRT 兼容的模型，跳过 ONNX 中间步骤

## 2.6 一句话总结

> **ORT→TRT 转换时的 Top-K 问题** = TRT 要求 K 值在编译时固定 + 排序行为明确，而 ONNX 模型往往带有动态 K 值的 Top-K 实现，两者不兼容。最实用的解法是把 Top-K 作为后处理步骤移出 TRT 引擎。

# 三、YOLO 系列演进对比（v8 → v11 → v26）

## 3.1 总览：三代架构的定位

| 版本 | 发布年份（约） | 定位 | 核心创新方向 |
|------|-------------|------|------------|
| **YOLOv8** | 2023 | 成熟稳定版 | Anchor-Free 化、C2f 结构、TaskAlignedAssigner |
| **YOLOv11** | 2024 | 效率优化版 | C3k2 模块、C2PSA 注意力、SPPF 改进、推理加速 |
| **YOLO26** | 2025+ | 端到端革新版 | 去除 NMS、去除 DFL、End-to-End 检测 |

> YOLO 版本编号本身是线性的，但 v11 到 v26 之间存在多个中间版本迭代（v12~v25），这里只聚焦三个标志性版本。

## 3.2 详细对比

| 维度 | YOLOv8 | YOLOv11 | YOLO26 |
|------|--------|---------|--------|
| **Backbone** | C2f（跨阶段部分连接 × 2，更丰富梯度流） | C3k2（C2f 的变体）+ C2PSA（注意力增强） | 进一步轻量化 |
| **Neck** | FPN + PAN（常规特征金字塔） | SPPF 改进版 | 架构升级，多尺度融合优化 |
| **Head** | Anchor-Free + 解耦头（分类/回归分离） | Anchor-Free + 解耦头 | End-to-End Head（无 NMS） |
| **正负样本分配** | TAL（Task-Aligned Assigner） | TAL | STAL（小目标感知标签分配） |
| **Loss** | DFL + CIoU | DFL + CIoU | ProgLoss（渐进式损失平衡） |
| **NMS 后处理** | 需要 NMS | 需要 NMS | 不需要 NMS（End-to-End） |
| **优化器** | SGD / AdamW | SGD / AdamW | MuSGD（融合动量的 SGD） |
| **主要优势** | 成熟稳定，生态完善 | 推理速度更快 | 端到端部署更简洁 |

## 3.3 关键概念详解

### 3.3.1 Anchor-Free（无锚框检测）

**原来的方式（Anchor-Based）**：在特征图每个位置预先放置大量预设尺寸的锚框（Anchor Box），训练模型预测锚框的偏移量。问题是锚框的超参数（大小、比例）对检测效果很敏感，需要针对数据集精心设计。

**YOLOv8 的方式（Anchor-Free）**：不再预设锚框，而是让模型直接预测目标的**中心点** + **宽高**。这种方式：
- 减少了超参数调优的工作量
- 对目标尺度变化适应更好
- 降低了计算复杂度

> 一句话：**从 anchor to point**，从预测框偏移到预测点坐标。

### 3.3.2 C2f（YOLOv8 Backbone）

C2f 是对 CSPNet（Cross Stage Partial Network）的改进：

```
传统 CSP 结构：
  输入 → Conv → Split → [分支1 Conv] + [分支2 Conv] → Concat → Conv → 输出

C2f 结构：
  输入 → Conv → Split → [分支1 Conv] + [多个 Bottleneck] → Concat → Conv → 输出
优化点：用更多的 split + concat，减少 Conv 数量，丰富梯度流路径
```

C2f 与 C3 相比，梯度流更丰富（更多的 split 和 concat 分支），意味着模型可以学到更多样化的特征，同时控制了参数量。

### 3.3.3 C3k2（YOLOv11）

C3k2 是 C2f 的进一步优化变体：

```
C3k2 → 将 C2f 中的 Bottleneck 替换为 C3k 模块
C3k → 带有内核大小可配置的卷积，灵活性更高
```

相比 C2f，C3k2 在保持参数量相近的情况下，通过更灵活的内核大小配置，提升了特征提取能力。

### 3.3.4 C2PSA（YOLOv11）

**PSA** = **P**artial **S**elf-**A**ttention（部分自注意力）

```
C2PSA → 在 C2 结构中融入自注意力机制
        让模型在空间维度上关注更重要的区域
        可以理解为：CNN + Attention 的混合模块
```

自注意力机制最初来自 Transformer，C2PSA 将其以轻量化的方式嵌入 YOLO 的 C2 结构中，在不显著增加计算量的前提下增强了特征表达能力。

### 3.3.5 SPPF（Spatial Pyramid Pooling - Fast）

| 概念 | 作用 |
|------|------|
| **SPP**（原始） | 用不同尺寸的池化核提取多尺度特征，增加感受野 |
| **SPPF**（YOLOv8 改进） | 用串联的小核（5×5）替代并联的大核，**更快** |
| **SPPF 改进版**（v11） | 进一步优化通道数或结构，提升效率 |

SPPF 让模型在**不增加输入分辨率**的情况下，获得更大的感受野，对小目标和多尺度目标检测更友好。

### 3.3.6 DFL（Distribution Focal Loss）

**DFL** 的全称是 **Distribution Focal Loss**。

**核心思想**：传统的边界框回归直接预测一个具体值（如 x 坐标是 122.5 像素），DFL 改为预测**一个分布**（比如 x 坐标有 20% 概率是 122，75% 概率是 123，5% 概率是 124）。

```
传统回归：预测值 = 122.5（直接回归）
DFL 回归：分布 = [0, 0, ..., 0.2, 0.75, 0.05, ..., 0]
                    ↑      ↑        ↑
                   120    122      123    124
```

**优点**：
- 边界框定位更**鲁棒**（不依赖单一数值）
- 对模糊边界有更好的**不确定性建模**
- 提升小目标边界框回归精度

**在 YOLO26 中移除 DFL**：YOLO26 选择去除 DFL，原因是为了降低计算复杂度和推理延迟，配合 End-to-End 架构做整体简化。

### 3.3.7 TAL（Task-Aligned Assigner）

**TAL** = **T**ask-**A**ligned **A**ssigner（任务对齐分配器）

**要解决的问题**：在目标检测中，一个 anchor 需要同时完成两个任务——分类（判断是什么物体）和回归（判断框在哪）。传统做法中，正负样本分配只考虑 IoU（回归质量），忽略了分类质量。

**TAL 的做法**：综合考虑**分类分数**和**回归质量**（IoU），为每个目标分配最匹配的 anchor。

```
TAL 对齐指标 = (分类分数)^α × (IoU)^β
其中 α, β 为超参数，控制两者权重
```

**效果**：更精准的正负样本分配 → 更高质量的检测结果。

### 3.3.8 STAL（Small Target-Aware Label Assignment）

**STAL** = **S**mall **T**arget-**A**ware **L**abel **A**ssignment（小目标感知标签分配）

**YOLO26 引入**，TAL 的专项改进版：

| 维度 | TAL（v8/v11） | STAL（v26） |
|------|--------------|------------|
| 分配策略 | 全局统一分配 | 针对小目标有单独的策略 |
| 对小目标的处理 | 容易被大目标抢占正样本 | 小目标有更高的分配优先级 |
| 匹配阈值 | 固定 | 与目标尺寸自适应 |

**背景**：小目标本身像素少、特征弱，在传统的 TAL 中容易被大目标"抢走"正样本，导致小目标漏检严重。STAL 通过**尺寸自适应的匹配策略**，让小目标更容易分配到正样本进行训练。

### 3.3.9 ProgLoss（Progressive Loss）

**ProgLoss** = **Prog**ressive **Loss**（渐进式损失平衡机制）

**要解决的问题**：YOLO 训练过程同时优化多个损失项（分类损失、回归损失、DFL 损失等），不同损失项的权重分配对训练效果影响很大。传统做法是手动调权重，或者使用固定权重。

**ProgLoss 的做法**：在**训练过程中动态调整**各个损失项的权重。

```
训练前期：回归损失权重高 → 先学会"框在哪里"
训练中期：分类损失权重上升 → 再学会"是什么物体"
训练后期：平衡收敛 → 精细调优
```

**优点**：
- 减少手动调参的工作量
- 训练过程更稳定
- 最终精度更高

### 3.3.10 MuSGD（Momentum-incorporated SGD）

**MuSGD** = **Mu**mentum-incorporated **SGD**（融合动量的随机梯度下降）

YOLO26 引入的优化器，结合了两种方法的优点：

| 方法 | 特点 | 不足 |
|------|------|------|
| **SGD**（随机梯度下降） | 泛化能力强，不易过拟合 | 收敛速度慢，容易震荡 |
| **Adam / AdamW** | 自适应学习率，收敛快 | 泛化能力不如 SGD |
| **MuSGD** | 两者兼顾 | — |

**本质**：在 SGD 中加入动量项（momentum），既保留了 SGD 较强的泛化能力，又获得了类似 Adam 的收敛稳定性。是一种务实且有效的优化器选择。

### 3.3.11 End-to-End 检测（去除 NMS）

**传统检测流程**：

```
模型输出 → 大量候选框 → NMS 筛选（去除冗余框） → 最终结果
```

**NMS 的问题**：
- NMS 是一个超参数敏感的后处理步骤（IoU 阈值、置信度阈值）
- 增加推理延迟
- 难以在端到端训练中联合优化

**YOLO26 的做法**：

```
模型输出 → 没有冗余框 → 直接输出最终结果（跳过 NMS）
```

**实现方式**：YOLO26 通过改进的训练策略（如 One-to-One 匹配）和 Loss 设计，让模型在训练过程中学会只输出一个最佳预测框，而不是输出一堆框然后靠 NMS 去重。

**优势**：
- 部署更简洁（少一个后处理步骤）
- 推理更快
- 消除 NMS 阈值调优的工作

## 3.4 关于 P2 / P6 的概念

### 3.4.1 什么是 P2 / P6

P2 和 P6 是 YOLO 特征金字塔中的不同层级，对应主干网络不同阶段的输出特征图：

| 术语 | 层级位置 | 分辨率（相对原图） | 感受野 | 特征类型 |
|------|---------|-------------------|--------|---------|
| **P2** | 早期层（浅层） | 高（1/4 下采样） | 小 | 细节/几何信息（边缘、纹理、位置） |
| **P6** | 极深层（深层） | 极低（1/64 下采样） | 极大 | 高级语义信息（类别、上下文） |

```
输入 → P1 → P2 → P3 → P4 → P5 → P6
       低层(高分辨率,浅层语义)    高层(低分辨率,深层语义)
       小目标检测                大目标检测
```

### 3.4.2 直觉困惑：P6 层为何能检测小目标？

一个常见的直觉困惑是：

> "P6 层是网络的极深层，特征图分辨率极低，每个像素点的感受野巨大，包含的是高级语义信息。按理说小目标的像素信息早已被'淹没'，为什么 P6 层的加入反而能提升小目标检测能力？"

这个直觉是对的——**P6 层本身确实不直接检测小目标**。它能帮助检测小目标，靠的是 **特征金字塔网络（FPN）** 的巧妙设计。

### 3.4.3 FPN 核心机制：自上而下的特征融合

FPN（Feature Pyramid Network）像一个"广播网络"，让高低层特征互相沟通：

```
自底向上（主干网络前向传播）：
  P2(高分辨率) → P3 → P4 → P5 → P6(低分辨率)

自顶向下 + 横向连接（特征融合）：
  P6(语义) → 上采样 → 与 P5 融合 → 再上采样 → 与 P4 融合 → ... → 与 P2 融合
```

具体流程分三步：

1. **自底向上**：主干网络前向传播，产生从高分辨率（P2）到低分辨率（P6）的一系列特征图
2. **自顶向下 + 横向连接**：高层特征（如 P6）**上采样**放大到与下一层相同尺寸，然后与对应层的特征图**逐元素相加融合**
3. **产生新特征**：融合后的特征图既包含高层语义信息（知道"是什么"），又保留了浅层位置细节（知道"在哪里"）

### 3.4.4 P6 与 P2 的分工与协作

| 环节 | P6 的核心任务 | P2/P3 的核心任务 |
|------|-------------|-----------------|
| **提供什么** | 提供"大局观"和"语义指引"（告诉模型这片区域大概率是什么类型的物体） | 提供"高分辨率细节"（告诉模型目标的边缘、轮廓在哪里） |
| **负责检测** | 不负责检测小目标（它的检测头只检测大目标，因为原生特征图太小） | 负责最终输出小目标的框和类别（在增强后的特征图上进行回归和分类） |
| **如果没有对方** | 如果只有 P6，小目标直接"消失"在网格里 | 如果只有 P2，误报极多（把纹理误判为缺陷，缺乏全局上下文） |

> **关键理解**：最终用于检测小目标的那个"P2"，实际上是 **"加载了 P6 语义知识的增强版 P2"**。它既保留了高清细节（能看见小点），又获得了高层语义（知道这个小点像什么）。

### 3.4.5 一个形象的类比

| 角色 | 对应层级 | 能力 |
|------|---------|------|
| **高级军官（P6）** | 高层语义 | 不亲自射击（检测小目标），但向下级提供精确的坐标和敌军描述 |
| **前线士兵（P2）** | 低层细节 | 拥有高精度步枪（高分辨率），负责执行射击 |

P6 提供"认知"，P2 提供"视力"，两者融合后由 P2 负责"扣动扳机"（执行检测）。

### 3.4.6 FPN 融合的本质

原始的 P2（没有融合） vs 增强后的 P2（带有横向连接和自上而下的路径）：

- **原始 P2**：只包含边缘、纹理等低级特征，缺乏语义信息，误报多
- **增强 P2**：融合了 P6 下传的语义先验知识，能区分"有意义的纹理"和"噪声"
- **P6 的作用不是被动观看，而是主动的"信息注入"**——P2 在融合后已经被 P6 "改造"了

### 3.4.7 实际意义：加 P2 / 加 P6

| 操作 | 目的 | 效果 |
|------|------|------|
| **加 P2 进 Neck** | 把小物体的特征信息引入特征金字塔 | 提升小目标检测能力 |
| **加 P6 扩展** | 增加一个更大的下采样层级（64 倍），配合更大分辨率输入 | 处理超大分辨率输入，为整个金字塔提供更强的语义指导 |

# 四、PLC（可编程逻辑控制器）

> **来源**：整理自《疑难杂症》笔记中的 PLC 相关问答
> **用途**：记录 PLC 基础知识及其与 Vision Master / 机械手的协同工作方式

---

## 4.1 官方定义

**PLC** 的全称是 **Programmable Logic Controller（可编程逻辑控制器）**。它是一种专为工业环境设计的数字运算操作电子系统，可以看作是一台极其稳定、不怕灰尘油污、专门用来控制机器的"加固型电脑"。

**核心定位**：工业自动化中不可或缺的控制核心。视觉系统负责"看"（Seeing），机器人负责"做"（Acting），而 PLC 负责"思考/协调"（Orchestrating）——它处理逻辑、I/O 信号，并触发视觉系统或机器人。

## 4.2 PLC 具体做什么

PLC 不会去处理复杂的图像像素（那是视觉系统的事），它主要处理两件事：**逻辑运算** 和 **输入/输出（I/O）控制**。

| 环节 | 说明 | 示例 |
|------|------|------|
| **接收信号（输入）** | 接收按钮、传感器、限位开关以及视觉系统发来的结果 | "产品 OK/NG" 或 "X/Y 坐标数据" |
| **执行程序（逻辑）** | 内部运行工程师编写的梯形图程序，根据输入信号做出判断 | "如果视觉给坐标，且机械手在位，则启动移动" |
| **发送指令（输出）** | 控制继电器、气缸、电机驱动器以及给机械手发送运动指令 | 控制传送带启停、触发抓取动作 |

## 4.3 在"视觉 + 机械手"场景中 PLC 扮演的角色

在 "相机定位引导机械手抓取" 系统中，PLC 是连接视觉和机械手的 **"桥梁"**，典型流程如下：

```
产品到位 → PLC 触发拍照 → Vision Master 计算坐标
                                      ↓
机械手抓取 ← PLC 转发坐标 ← PLC 检查坐标是否合理
```

1. **触发拍照**：PLC 检测到产品到位，发送信号给 Vision Master（视觉软件）命令相机拍照。
2. **接收坐标**：Vision Master 计算完像素偏移并转换成物理坐标后，通过以太网或串口将 (X, Y) 坐标数据发送给 PLC。
3. **逻辑防呆**：PLC 接收后会检查坐标是否在合理范围内（防止视觉误判导致撞机）。
4. **引导执行**：PLC 将坐标通过特定通讯协议（如 EtherCAT、Modbus）转发给机械手控制器。

> **简单总结三者的分工**：
> - 相机（眼）——看到在哪
> - PLC（脑）——判断要不要抓、怎么通知
> - 机械手（手）——执行抓取

## 4.4 PLC 的主要特点

| 特点 | 说明 |
|------|------|
| **皮实耐用** | 能在零下几十度到零上几十度、充满灰尘和震动的工厂里稳定运行十几年 |
| **实时性强** | 扫描一个循环通常只需几毫秒，确保急停按钮按下时瞬间切断电源 |
| **编程直观** | 主要使用梯形图（Ladder Diagram），看起来像电路图，电气工程师容易上手 |
| **模块化** | 可根据需要扩展 I/O 模块、通讯模块等 |

## 4.5 常见品牌

| 系列 | 品牌 |
|------|------|
| **欧系** | 西门子（Siemens，行业老大）、倍福（Beckhoff，常配 EtherCAT 总线） |
| **日系** | 三菱（Mitsubishi）、欧姆龙（Omron）、基恩士（Keyence） |
| **美系** | 罗克韦尔（Rockwell / AB） |

## 4.6 常见误区纠正

### 4.6.1 它是"品类"代称，不是"系列"代称

PLC 是一类 **设备** 的统称，就像"智能手机"一样——华为、苹果、小米都叫智能手机，但它们是不同的品牌和系列。

- PLC 是**硬件**：它是一块实实在在的电路板（带有 CPU、内存、输入输出接口），是看得见摸得着的"铁疙瘩"。
- 它不是"作用于"硬件，它"**本身就是**"硬件。它不像 Windows 系统那样装在电脑里；开机就直接运行内部固件，不需要外挂操作系统启动。

### 4.6.2 关于"直接操作硬件"

PLC 通过物理 I/O（数字/模拟输入输出）直接连接到按钮、传感器、电机和阀门。它不通过 Windows 等操作系统进行抽象，而是**实时循环（扫描周期）直接读取输入并设置输出**。

但 PLC 内部运行的不是"系统"（如 Windows），而是 **"逻辑程序"**（梯形图）。它更像是一个 **"硬件逻辑执行器"**。

### 4.6.3 可以把 PLC 想象成一个"超级智能继电器盒子"

在 PLC 发明之前，工厂控制一台机器要用几十上百个物理继电器搭接逻辑，改一次工艺得把电线全拆了重焊。

**PLC 的本质：用软件逻辑代替物理电线。** 改变内部的一段梯形图程序，就相当于改了电路板上的接线，从而改变了硬件端子的通断关系。

> **一句话总结**：PLC 是一类工业专用的硬件控制器，它本身带有物理接线端子，能直接驱动外部设备（电机、气缸、指示灯），但具体如何驱动，取决于工程师下载到它内部的逻辑程序（梯形图）。

# 五、HALCON 文件体系（.hdev / .hdvp / .hdpl）

> **来源**：整理自《疑难杂症》笔记中的 HALCON 文件类型说明
> **用途**：厘清 HALCON 开发中最核心的三种文件类型及其协作关系

---

## 5.1 概述

在 HALCON 开发中，`.hdev`、`.hdvp`、`.hdpl` 是最核心、最常用的文件类型。它们的关系类似于"可执行的完整项目"和"可复用的功能模块"。

## 5.2 .hdev —— 主程序文件（可执行入口）

**本质**：HDevelop 的主程序文件，整个项目的入口。

| 特性 | 说明 |
|------|------|
| **内容** | 包含 `main()` 级别的执行流程（for 循环、读取图像、调用算法） |
| **本地函数** | 可在文件内部定义本地函数（Local Procedures），但只能在该 `.hdev` 内部使用 |
| **运行方式** | 可直接运行（点击 HDevelop 的"运行"按钮），是整个程序的启动点 |

## 5.3 .hdvp —— 外部函数文件（可复用的子程序）

**本质**：HDevelop 的外部函数文件，只能包含一个函数（Procedure），不能包含主程序流程。

| 特性 | 说明 |
|------|------|
| **参数接口** | 必须有明确的输入、输出参数定义 |
| **运行方式** | 不能独立运行，必须被 `.hdev` 或其他 `.hdvp` 调用 |
| **复用方式** | 保存为 `.hdvp` 后，会出现在 HDevelop 的算子列表中，像内置算子一样随时调用 |

## 5.4 .hdpl —— 库函数文件

当有多个 `.hdvp` 函数时（如 `Usb_Process.hdvp`、`Detect_Defect.hdvp`），可以把它们打包成一个 **`.hdpl`（库函数文件）**。调用时只需 import 一个文件，就能获得一整套功能模块。

## 5.5 核心区别与协作

| 对比维度 | .hdev（主程序） | .hdvp（外部函数） | .hdpl（库） |
|---------|---------------|-----------------|------------|
| **定位** | 项目入口，负责调度 | 功能模块，负责具体算法 | 多个功能的集合 |
| **能否直接运行** | ✅ 可以 | ❌ 必须被调用 | ❌ 必须被调用 |
| **包含内容** | 主流程 + 多个本地函数 | 只能包含 1 个函数 | 多个 `.hdvp` 的打包 |
| **作用范围** | 函数默认仅限本文件 | 全局可用（像自带算子） | 全局可用 |
| **参数接口** | 无固定要求 | 必须有输入/输出参数定义 | 统一导入 |

## 5.6 典型用法

| 阶段 | 做法 |
|------|------|
| **开发阶段** | 核心算法写在 `.hdvp` 中，在 `.hdev` 中写循环调用、传参、展示结果 |
| **部署阶段** | 将调试好的 `.hdvp` 发给现场工程师，在各自的 `.hdev` 中调用 |
| **库管理** | 多个 `.hdvp` 打包为 `.hdpl`，只需 import 一个文件即可使用整套模块 |

> **示例**：在 `.hdev` 中通过 `import` 语句引用外部函数：
> ```
> import 'D:/项目/孔洞检测/Usb_Process.hdvp'
> ```

# 六、九点标定核心问答

> **来源**：整理自《疑难杂症》中的标定相关问答
> **要点**：X/Y优先、基准点公式、眼在手外 vs 眼在手上

---

## 6.1 X优先 / Y优先

在九点标定中，**X优先** 和 **Y优先** 决定机构走"弓"字形路径时，先走水平方向还是垂直方向。

| 模式 | 路径（3×3 九宫格） | 特点 |
|------|-------------------|------|
| **X优先** | 1→2→3 → 6→5→4 → 7→8→9 | 先沿 X 轴扫完一行，换行 |
| **Y优先** | 1→4→7 → 8→5→2 → 3→6→9 | 先沿 Y 轴扫完一列，换列 |

**核心原则**：哪个轴运动更稳定、精度更高，就让哪个轴优先。非优先轴在单行/单列内保持不变。

## 6.2 图片顺序与优先匹配

如果采集顺序是"上左→上中→上右→中右→中中→中左→下左→下中→下右"（"S"形路径），应设置为 **X优先**。

判断方法：看 **优先轴（先变的那个轴）**——本例中 X（水平）在每行内连续变化，Y（垂直）仅换行时变化，符合 X 优先规律。

> 注意：软件中点的编号顺序（填表顺序）必须与实际物理移动顺序**完全一致**，否则标定结果会错。

## 6.3 相机固定不动（眼在手外）

| 配置 | 相机位置 | 标定特点 | 公式 |
|------|---------|---------|------|
| **眼在手外** ✅ | 固定安装，不动 | 映射关系唯一固定，标定一次管很久 | 目标点 = 基准点 + 偏移量 |
| **眼在手上** ❌ | 装在机械臂上 | 每次拍照位置不同，需旋转中心标定 | 需要复杂坐标变换 |

**相机不动（眼在手外）带来的便利**：
- 像素与物理坐标的映射关系（XML 文件）**永久有效**，只要不碰相机就不用重新标定
- 视野中心成为**绝对的空间锚点**，标定时第 5 点（中心点）的机器人坐标可直接作为基准点
- 生产时：`目标点 = 基准点(第5点机器人坐标) + 视觉计算的偏移量`

> **一句话**：只要相机没被碰歪，基准点就是金标准，永远不用改。

# 七、Blob 分析

> **来源**：整理自《疑难杂症》中的 Blob 分析原理说明

---

## 7.1 什么是 Blob

**Blob** = **B**inary **L**arge **Ob**ject（二进制大对象），指图像中灰度值相近且彼此连通的像素区域。在工业场景中，一个焊点、一颗螺丝、一处划痕、一个印刷标记，都可以看作一个 Blob。

## 7.2 工作原理

Blob 分析遵循标准化的三步流程：

### ① 图像分割（二值化）

设定灰度阈值，将灰度图转为黑白二值图，目标变为白色前景，背景变为黑色。

| 阈值方式 | 说明 |
|---------|------|
| **单阈值** | 设定一个阈值范围（如 100~255），提取该区间的像素 |
| **双阈值** | 同时提取两个独立的灰度区间，适合多层目标 |
| **自动阈值** | 根据灰度直方图自动计算阈值，适合光照稳定的场景 |

### ② 连通域分析

识别所有彼此连通的白色像素区域，按连通标准分两种：

| 连通方式 | 方向 | 适用 |
|---------|------|------|
| **4 连通** | 上下左右 | 速度快，适合规则目标 |
| **8 连通** | 上下左右 + 对角线 | 能找到更完整的目标区域 |

### ③ 特征提取

每个 Blob 自动计算以下特征：

| 特征类别 | 指标 | 用途 |
|---------|------|------|
| **基础特征** | 面积、周长、质心 | 判断有无、定位 |
| **形状特征** | 外接矩形长/短轴、圆形度、矩形度 | 区分目标形状是否异常 |
| **高级特征** | 孔洞数量与面积、拓扑关系 | 检测内部缺陷 |

## 7.3 应用场景

| 场景 | 做法 | 优点 |
|------|------|------|
| **缺陷检测** | 分析 Blob 面积/形状是否异常 | 无需模板，灵活 |
| **目标计数** | 统计 Blob 数量 | 快速可靠 |
| **定位测量** | 输出 Blob 质心和尺寸 | 精度可达亚像素级 |

## 7.4 使用限制

- 仅处理 8 位灰度图，彩色图需先转灰度
- 要求背景纯净、目标与背景对比鲜明
- 目标相互重叠时会误识别为一个整体
